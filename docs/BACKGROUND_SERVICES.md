# Background services

Which work runs where: in-process threads, Celery workers, beat schedules, and the realtime event stream.

## Threads vs Celery

| Work | Runs on | Needs a worker? |
| --- | --- | --- |
| Interactive generation jobs | Daemon thread in django (`dispatch_job`) | No |
| Interactive analysis runs | Daemon thread in django (`runner.dispatch`) | No |
| Automation pipeline runs | Celery `run_pipeline_task` (thread fallback if broker is down) | Preferably |
| Automation schedules | Celery beat `run_scheduler_tick` (or a management command) | Yes |

The trade-off of the thread model: the stack is fully functional without any Celery worker, but **in-flight interactive runs die if the django container restarts** — they resurface as stuck `running` rows, not as retried jobs.

## Celery services

Three compose services share the django image (`docker-compose.local.yml`):

- `celeryworker` — executes automation tasks. Broker and result backend are Redis (`REDIS_URL`).
- `celerybeat` — periodic scheduler, using `django_celery_beat`'s database scheduler, so periodic tasks are editable in the Django admin.
- `flower` — task dashboard on :5555, credentials generated by `just bootstrap`.

## The automation scheduler

Cron [schedules](/docs/AUTOMATION_GUIDE) fire when something calls `scheduler.tick()`, which checks for enabled schedules whose `next_run_at` has passed. Two ways to make that happen:

- **Celery beat** running the `run_scheduler_tick` task — the production recommendation.
- `just manage runautomationscheduler` — a standalone loop that ticks every 30 seconds, for setups without beat.

If neither runs, schedules simply never fire; pipelines can still be triggered manually.

## Realtime events (SSE)

Progress updates reach the browser over server-sent events, not polling:

- Publishers call `backend/realtime/events.publish(channel, event)`, which pushes JSON into Redis pub/sub. Channels: `generation:<job_id>`, `analysis:<run_id>`, `runtime:<container_id>`, and the `dashboard` broadcast.
- `GET /api/realtime/stream?channels=generation:<id>,analysis:<id>` is a plain Django streaming view mounted outside ninja (`config/urls.py`) because ninja can't stream. It requires an authenticated session and at least one channel.

## Runtime housekeeping

Two management commands keep the container fleet honest, useful from cron in production:

- `just manage sync_container_status` — reconciles DB container state with what Docker actually reports.
- `just manage reconcile_app_routes` — rewrites the per-app Traefik route files (see [Deployment guide](/docs/deployment-guide)).
