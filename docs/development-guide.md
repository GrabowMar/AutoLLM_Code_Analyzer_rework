# Development guide

Day-to-day workflow: just targets, tests, environment files, and how the backend and frontend are laid out.

## Toolchain

Python 3.13 with [uv](https://docs.astral.sh/uv/) (`pyproject.toml` + `uv.lock`), but you rarely need either on the host — every command runs inside Docker Compose. The `justfile` pins `COMPOSE_FILE=docker-compose.local.yml`, so `just` targets always hit the local stack.

## just targets

| Target | What it does |
| --- | --- |
| `just bootstrap` | Create `.envs/.local/.django` and `.postgres` from templates, generate secrets |
| `just build [args]` | `docker compose build` |
| `just up` / `just down` | Start / stop the stack |
| `just prune [-v]` | Remove containers **and volumes** (full reset) |
| `just logs [service]` | Follow container logs |
| `just test [args]` | `docker compose run --rm django pytest` — same command as CI |
| `just manage <args>` | Run any `manage.py` command (migrate, shell, seeds, …) |
| `just frontend-install` | `npm install` inside the frontend container |
| `just frontend-dev` | Run the frontend dev server in the foreground |
| `just frontend-build` | Production frontend build |

## Environment files

Local settings come from `.envs/.local/.django` (plus `.postgres` for DB credentials):

- `DJANGO_DOMAIN` — used in `DEFAULT_FROM_EMAIL` and `ALLOWED_HOSTS` defaults.
- `REDIS_URL` — Celery broker/result backend and cache.
- `CELERY_FLOWER_USER` / `CELERY_FLOWER_PASSWORD` — Flower dashboard login (generated by bootstrap).
- `OPENROUTER_API_KEY` — optional global fallback key for users without a personal key.
- `OPENROUTER_ALLOW_GLOBAL_KEY_FALLBACK` — set `False` to force every user to bring their own key.
- `FRONTEND_PUBLIC_ORIGIN` — the URL users open in the browser; used in verification/reset emails.
- `DJANGO_ALLOWED_HOSTS`, `CORS_ALLOWED_ORIGINS`, `CSRF_TRUSTED_ORIGINS` — set all three when serving from anything other than localhost.

## Backend layout

Django 6 project; settings in `config/settings/{base,local,production,test}.py`, the API root in `config/api.py`. Fifteen apps under `backend/`:

| App | Responsibility |
| --- | --- |
| `users`, `tokens`, `credentials` | Accounts (allauth headless + MFA), Bearer API tokens, encrypted per-user provider keys |
| `llm_models` | Model catalog synced from OpenRouter |
| `generation` | Generation jobs, templates, prompt assembly, OpenRouter client |
| `analysis` | Analyzer workspaces, tool catalog, runs, findings |
| `statistics`, `rankings`, `reports` | Aggregation over findings and jobs, model scoring, saved reports |
| `runtime` | Builds and runs generated apps as Docker containers, routing/proxy |
| `realtime` | Server-sent events stream backed by Redis pub/sub |
| `export` | CSV/JSON/SARIF downloads |
| `docs`, `system` | This docs viewer; staff-only system snapshots |
| `automation` | Pipelines, batches, schedules |

Apps follow one convention: `models.py`, `api/{views,schema}.py`, `services/`, `tasks.py`, `tests/`. Views raise ninja's `HttpError`; services stay free of ninja imports and raise `backend.common.exceptions.*`, which an exception handler in `config/api.py` maps to HTTP responses.

Scoring constants live in one place: `backend/rankings/services/constants.py` defines `SEVERITY_WEIGHTS` (shared by rankings, statistics, and report generators) and the `SEVERITY_WEIGHT_SCHEMES` used by the sensitivity analysis — change them there, nowhere else. The methodology itself is described in [Architecture](/docs/ARCHITECTURE).

## Frontend layout

SvelteKit with Svelte 5 runes, Tailwind 4, in `frontend/`. Routes live under `src/routes/(app)/` (one directory per feature: `generation`, `analysis`, `automation`, `rankings`, …) plus auth pages. API clients are in `src/lib/api/`. In the local stack the dev server on :8000 proxies API calls to django via `API_TARGET=http://django:8000` — always develop against http://localhost:8000, not :8001.

## Testing

```bash
just test                        # whole backend suite
just test backend/analysis       # one app
just test -k workspace -x        # any pytest args pass through
```

pytest runs with `--ds=config.settings.test --reuse-db`. Add `--create-db` after changing seed data (e.g. the analyzer tool YAML), since the reused test DB keeps the old catalog. Shared fixtures live in `backend/conftest.py`; per-app factories in `backend/<app>/tests/factories.py`.

Lint/format is ruff + mypy (backend), `svelte-check` (frontend), wired into `.pre-commit-config.yaml`.

## Working on these docs

The in-app viewer at `/docs` serves raw Markdown straight from the repo's `docs/` directory — no build step. Rules, enforced by how `backend/docs/services.py` and the client renderer work:

- Exactly one `# H1` as the first line; it becomes the page title. The first plain paragraph after it becomes the landing-card blurb.
- No YAML frontmatter (it would render as content). No images (nothing serves them).
- Cross-link other pages as absolute `/docs/<slug>` paths — relative `.md` links don't resolve in the app.
- Mermaid fences and `> [!NOTE]` / `> [!WARNING]` alerts are supported; code fences highlight for the common languages (bash, python, ts, json, yaml, dockerfile, …).
- New top-level files land in the "Other" category unless you add their slug to `CATEGORY_MAP` in `backend/docs/services.py`.
